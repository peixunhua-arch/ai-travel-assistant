## 4. 开发侧「常驻原则」提示词（来自 UX 大纲）

写功能、改 UI 时默认遵守：

1. **少打字**：结构化表单 + 快捷 chip，自然语言作补充  
2. **可预期**：每步知道在等什么、等多久、失败怎么办  
3. **可扫读**：行程用卡片/时间轴，不用大段文字  
4. **可延续**：生成 → 保存 → 打开 → 评价，链路不断  
5. **可信任**：AI 免责、真实 POI 来源、评价如何影响推荐要看得见  

架构级决策：

- **闲聊 vs 规划分离**（聊天走 `/api/chat`，规划走表单 + `/api/trip/generate`）
- Tab 目标形态：**规划 / 行程 /（社区）/ 我的**；中间「+」可放 AI 聊天
- Capability：**能**签证/天气/美食问答、表单规划、保存评价；**不能**订票付款、实时导航、保证 100% 准确

## 5. 代码内 LLM Prompt（真正喂给模型的）

文件：`apps/server/src/services/claude.ts`、`apps/server/src/store.ts`

### 5.1 闲聊 `SYSTEM_PROMPT`

```
你是一个专业、热心的中文旅游助手。回答要简洁实用，重点突出。
涉及签证、天气、安全等信息时，提醒用户以官方渠道为准。
```

用于：`ask` / `askStream`（SSE）。

### 5.2 行程规划 `TRIP_SYSTEM_PROMPT`（核心）

人设 + 硬约束摘要：

| 块 | 要点 |
|----|------|
| 人设 | 专业中文行程规划师；按目的地/天数/预算/偏好出可执行每日行程 |
| 工具 | **必须先** `get_weather`、`search_place`；非交通项必须带真实 `poiId`；禁止编造店名 |
| 口碑 | `community.likeRatio` 高优先；负面 `topTags` 谨慎 |
| 输出 | **只输出 JSON**，无解释、无 \`\`\`json；含 destination / budgetEstimate / days[].theme/items |
| 内容 | 天数一致；每天 4–6 items；简体中文；**不要**输出 lat/lng/rating/address/links（后端回填） |

工具说明（给模型）：

- `get_weather`：城市未来 3 天天气，安排室内外  
- `search_place`：高德真实 POI，必须从结果选点并带 `poiId`

运行时收尾 / 纠错（拼进 user，非 system）：

- 收尾：工具数据已齐 → 从 POI 选点 → 直接输出最终行程 JSON  
- 重试：输出不是合法 JSON → 只输出符合要求的 JSON 对象  

### 5.3 用户消息模板（`generateTrip`）

拼装字段行：目的地、天数、预算、偏好、出行时间、同行人、节奏、出发城市、`补充要求：${req.prompt}`；登录用户再附加偏好段。

### 5.4 偏好注入（必须进 **user**，禁止进可缓存 system）

```
## 该用户的历史偏好（据其过往评价总结，请据此调整推荐）
- 偏好：…
- 避免：…（这些是他踩过雷的点，尽量别再推类似的）
```

硬约束：放进 user 消息，否则 Prompt Caching 永远 miss。

### 5.5 前端引导「伪 Prompt」（用户可见文案）

| 位置 | 内容 |
|------|------|
| 闲聊 chip | `日本要签证吗` / `周末周边游推荐` / `带老人旅行要注意什么` / `成都有什么好吃的` |
| 规划 chip | `帮我规划成都3天行程` |
| Onboarding | 随便聊 → 表单规划（真实天气+地点）→ 存行程+评价影响推荐 |
| 免责 | 行程由 AI 生成仅供参考；地点来自高德 POI |

## 6. 工程与阶段约束（写代码时的「系统提示」）

### 6.1 技术方案硬约束

- **密钥只在后端**；App 不碰 Claude / 网关 Key  
- **不以小红书/点评当数据源爬取**——只拼跳转；POI 走高德  
- Agent 工具轮最多约 10；收尾必须落最终 JSON；行程 `max_tokens` 宜大（如 8192）  
- 环境变量用 `CLAUDE_*` 前缀，避免被机器全局 `ANTHROPIC_*` 静默覆盖  

### 6.2 阶段目标速查

| 阶段 | 目标 |
|------|------|
| 0 | Monorepo / 环境 |
| 1 | `/api/chat` 闲聊 |
| 2 | `/api/trip/generate` 结构化 JSON |
| 3 | Agent + 高德/天气 + 地图外链 + 保存 |
| 3.5 | 评价能评/能存/能回显（再偏好与口碑反哺） |
| 4 | 编辑/分享/Caching/限流/合规/EAS |
| 4+/5 | 多模态、可交互 AI、长辈模式、主动助手 |

### 6.3 阶段 3.5 关键要点（`阶段3.5-关键要点.md`）

**本批做**：评价 → 落库 → 详情回显。  
**本批不做**（有数据后再接）：个人偏好进 prompt、社区口碑进搜索——后续已部分接通，改 prompt 时仍遵守「偏好不进 system cache」。

决策摘要：

1. DB 用 Node 内置 `node:sqlite`  
2. 行程混合：本地 AsyncStorage 为主，保存时上传拿 `serverTripId`  
3. 评价标签：赞「性价比高/值得去/本地人多/交通方便」；踩「太赶/踩雷/排队久/名不副实」
