## 1. 现状诊断

### 1.1 当前实现（代码，2026-07-10 核对 `apps/mobile/App.tsx`）

**已具备的基线（不要当成待办重复排期）：**

- 错误态已与 AI 回复分离：失败的用户消息下方独立显示「⚠️ 错误 + 点按重试」（`App.tsx` 的 `failed`/`handleRetry`），**不是**塞进 AI 气泡。
- AI 回复已走 Markdown 渲染（`src/Markdown.tsx`），标题/加粗/列表/表格可排版。
- 发送后已清空输入 + 禁用发送按钮防连点（`editable={!loading}` + 按钮 `disabled`）。

**仍存在的痛点：**

| 维度 | 现状 | 痛点 |
|------|------|------|
| 信息架构 | 单文件 `App.tsx`，无路由 | 聊天、规划、行程列表混在一起，用户找不到「我的行程」 |
| 对话 | 文本/Markdown 气泡 + 底部 loading | 无快捷入口；无长按复制；无空对话引导；无「新对话」 |
| 规划 | 与闲聊共用同一输入框 | 用户不知道该怎么「要一份行程」 |
| 等待 | 「AI 正在思考…」单行文案 | 行程生成可能 60s+，无进度、无取消、易以为卡死 |
| 引导 | 一句欢迎语 | 新用户不清楚能做什么 |
| 后端能力 | 仅 `/api/auth/device` + `/api/chat` | 行程、地图、评价等 UX 需提前设计，避免返工 |

### 1.2 目标形态（技术方案）

- 底部 Tab：**规划** / **行程** / **我的**
- 规划：结构化表单 → 生成 → **独立行程预览页**（非聊天气泡塞 JSON）
- 详情：按天时间轴 + 地图 + 跳转链接 + 👍/👎 评价
- 闭环：评价影响下次推荐（见技术方案 6.7）

## 2. 优化原则

1. **少打字**：结构化表单 + 快捷 chip，自然语言作补充
2. **可预期**：每一步知道在等什么、要等多久、失败怎么办
3. **可扫读**：行程用卡片/时间轴，不用大段文字
4. **可延续**：生成 → 保存 → 打开 → 评价，链路不断
5. **可信任**：AI 免责声明、真实 POI 来源、评价如何影响推荐要「看得见」

## 4. P1：信息架构与导航（阶段 1.5～2）

### 4.1 迁移到 Expo Router + 底部 Tab

| Tab | 路由 | 职责 |
|-----|------|------|
| 规划 | `(tabs)/index` | 闲聊 + 行程生成入口 |
| 行程 | `(tabs)/trips` | 已保存行程列表 |
| 我的 | `(tabs)/profile` | 设置、免责说明、dev 配置 |

**空状态：**

- 行程 Tab 无数据：插画 + 文案「还没有行程」+ 按钮「去规划」
- 我的 Tab：隐私说明、AI 免责声明入口

### 4.2 核心 UX 决策：闲聊 vs 生成行程分离

不要把两种意图混在同一个「发送」里：

| 模式 | 入口 | 接口 |
|------|------|------|
| **随便聊聊** | 对话输入框 + 发送 | `POST /api/chat` |
| **生成行程** | 结构化表单 +「生成行程」按钮 | `POST /api/trip/generate` |

### 4.3 行程规划表单（`TripGenerateForm`）

对应 API 契约（技术方案 5.4）：

| 字段 | 控件 | 必填 |
|------|------|------|
| `destination` | 文本输入 + 历史/热门城市 chip | ✅ |
| `days` | 步进器 1～14 | ✅ |
| `budget` | 数字输入或档位（经济/舒适/不限） | 可选 |
| `preferences` | 多选 chip：美食/自然/亲子/轻松/打卡… | 可选 |
| `prompt` | 单行补充说明 | 可选 |

**交互：**

- 主按钮文案用 **「生成行程」**，与聊天的「发送」区分
- 表单校验失败：字段级红字提示，不弹笼统 Alert
- 提交后进入全屏/Sheet Loading（见 6.1）

**为什么用表单而不是「AI 追问」：**

用户只说「成都3天」时缺预算/人数/带不带娃，AI 本可反问。但技术方案的 `/api/trip/generate` 是**无状态、一次成型**的（无多轮会话上下文）。取舍：

- **第一版：结构化表单一次收集**关键字段，用必填/默认值兜住信息缺口，**不做对话式追问**。目的地、天数必填；其余给合理默认（预算「不限」、偏好可空），避免「等用户补充」的多轮往返
- 表单说不清的（如「不想走太多路」）走 `prompt` 补充字段，一次带给后端
- **多轮追问（需 API 扩展）属阶段 4**，见 8.2；在此之前别在 UI 上做「AI 反问」的假象，否则和无状态后端对不上

### 4.4 生成结果页（`TripPreview`）

- 生成成功 → **独立页面**，不在聊天气泡里展示 JSON
- 顶部摘要：目的地、天数、预估预算
- 主体：`DayPlan` 按天折叠卡片
- 底部固定操作栏：
  - **保存行程**（主按钮）
  - **重新生成**
  - **修改需求**（回到表单并保留上次输入）
- [ ] **未保存离开拦截**：预览页有结果但未保存就返回 → 弹窗「行程未保存，是否离开？」

### 4.5 规划 Tab 布局方案（实现前定稿）

附录 A 的「聊天 + 表单同屏上下叠」在小屏上会很长，**实现前二选一**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A（推荐）** | 规划 Tab 内 **二级 Tab**：`[ 聊聊 ]` `[ 规划行程 ]` | 意图清晰、单屏不挤 | 多一层切换 |
| **B** | 主屏聊天，表单收进 **底部 Sheet**（点「规划行程」拉起） | 聊天为主时不占屏 | Sheet 与键盘交互要调 |

> 阶段 2 默认采用 **方案 A**；若产品更偏「聊天驱动规划」再考虑 B。

### 4.6 表单扩展字段（阶段 2.5，可选）

影响行程质量但第一版 API 可先忽略，**前端先收集、后端逐步接入**：

| 字段 | 控件 | 用途 |
|------|------|------|
| `travelMonth` / 出行日期 | 月份选择或「下月 / 国庆」 | 天气、旺季、室内/室外 |
| `companions` | 独自 / 情侣 / 亲子 / 带老人 | 节奏与景点类型 |
| `pace` | 轻松 / 适中 / 特种兵 | 比 `prompt` 写「不想走路」更结构化 |
| `departureCity` | 可选文本 | 大交通、首日安排 |

写入 `GenerateTripRequest` 扩展字段或暂塞 `prompt` 结构化前缀，见技术方案 API 演进。

### 4.7 「重新生成」与已保存行程的关系

- **预览页「重新生成」**：默认进入**新一次预览**，不自动覆盖已保存行程
- 用户从**已保存详情**点「重新生成」：生成成功后弹窗 **「覆盖原行程 / 另存为新行程」**
- 避免用户误以为「重新生成 = 自动更新列表里那条」

## 5. P1：行程详情与列表（阶段 2～3）

### 5.1 行程详情页 `trip/[id].tsx`

**单点卡片（`TripCard`）信息层级：**

```
┌─────────────────────────────────────┐
│ ┌────┐ 09:30  🍜 美食               │
│ │缩略│ 陈麻婆豆腐（青羊区总店）       │  ← 左侧 POI 缩略图（高德 photos）
│ │ 图 │ 高德 4.5 · 川菜 · ¥60/人      │
│ └────┘ 地址…                        │
│ Claude 推荐理由（1～2 行）            │
│ [📍导航] [🔴小红书] [🟠大众点评]      │
│ 👍 👎  [性价比高] [排队久] …         │  ← 阶段 3.5
└─────────────────────────────────────┘
```

- [ ] **POI 缩略图**：高德 POI 详情常带 `photos` 字段，`enrichTrip` 回填时顺手取第一张存进 item（如 `photoUrl`）。无图时用类型占位图（见下方类型色），不要留白框。旅游 App 没图会很干。

**交互：**

- [ ] 顶部 Day Tab 或锚点：Day1 / Day2 / Day3
- [ ] 卡片默认收起简介，点击展开
- [ ] `transport` 类型弱化样式，不上地图 marker
- [ ] 跳转外链前轻提示：「将打开第三方 App/网页」
- [ ] **类型区分用「颜色 + 图标」双编码**，不能只靠颜色（色盲用户无法区分绿/橙）。约定固定图标：景点 🏛️、美食 🍜、住宿 🏨、交通 🚗，与 9.1 的类型色配套使用

### 5.2 地图联动（阶段 3，`TripMap`）

- [ ] 详情页：上半时间轴 + 下半地图（或可拖拽分割）
- [ ] 点卡片 → 地图高亮 marker
- [ ] 点 marker → 列表滚动到对应卡片
- [ ] 当天点位按时间顺序连线（示意即可，不做导航级路径规划）

### 5.3 我的行程列表 `(tabs)/trips`

- [ ] 列表项：目的地、天数、预算、更新时间
- [ ] 点击进入详情；左滑删除 + 二次确认
- [ ] 下拉刷新
- [ ] 骨架屏加载
- [ ] **冷启动示例行程**：无保存数据时，除「去规划」外可提供 **只读示例**（静态成都 2 日 JSON），让新用户先看懂 UI 长什么样
- [ ] **置顶/收藏**（阶段 4）：即将出行的行程 pin 到列表顶部
- [ ] **搜索/筛选**（阶段 4）：按目的地搜索

### 5.4 行程手动编辑（阶段 4，对应 `PUT /api/trips/:id`）

> 技术方案有 `PUT /api/trips/:id`（改时间、删景点，**不重新调 AI**），但纯 AI 生成难免有不合心意的点，用户需要能手动微调。这是「可延续」原则的一环。

- [ ] 详情页右上角 **「编辑」** 进入编辑态（与浏览态区分，避免误触）
- [ ] 单点操作：
  - 删除某个 POI（二次确认；删到某天为空时提示「这天已空，是否删除整天」）
  - 改时间（时间选择器，非自由文本，避免格式乱）
  - 拖拽排序（同一天内调整顺序；`transport` 项可跟随或锁定）
- [ ] **保存/放弃**：编辑态底部固定「保存修改」+「放弃」；有改动未保存时返回要拦截确认
- [ ] 保存走 `PUT`，成功后回浏览态并 Toast「已更新」
- [ ] ⚠️ 编辑不触发重新回填坐标/链接：用户删点是删，加点这一版**不做**（加点需重新查高德，属重新生成范畴）——在 UI 上只提供「删/改时间/排序」，别给「+ 添加地点」按钮造成预期落空

### 5.5 行程可读性增强（阶段 3+）

让行程「像真行程」，而不只是 AI 文字列表：

- [ ] **点与点通勤**：卡片间显示「约 15 分钟车程 / 步行 800m」（高德距离 API，与 `TripMap` 同期）
- [ ] **当日总览条**：Day 顶部一行：共 N 站 · 预估花费 · 户外/室内占比（可从 type 统计）
- [ ] **天气条**：详情页顶部展示目的地未来 3 天天气（复用 `get_weather` 数据）
- [ ] **营业时间**：POI 有 `opentime` 时角标「注意营业至 21:00」
- [ ] **预算对比**：`budgetEstimate` 高于用户填写预算时轻提示「可能略超预算」

### 5.6 行程管理（保存链路）

- [ ] **重复保存提示**：同目的地+相近天数已存在 → 「已有类似行程，仍要保存？」
- [ ] **删除撤销**：删行程后 5 秒内 Toast「撤销删除」
- [ ] **未保存草稿**：生成成功未保存就杀进程 → 下次打开可选恢复（AsyncStorage 存最后一次 preview，7 天过期）

## 6. P1：等待、错误与离线（全链路）

### 6.1 长任务状态机（行程生成）

```
idle → submitting → tool_running → parsing → enriching → success
                  ↘ failed
```

**用户可见：**

| 阶段 | 文案示例 |
|------|----------|
| submitting | 正在提交你的需求… |
| tool_running | 正在查询天气和地点… |
| parsing | 正在安排每日行程… |
| enriching | 正在补充地址和链接… |

- [ ] 步骤指示器（3～4 步）或进度条
- [ ] **取消**按钮（`AbortController`）
- [ ] 超时 60s+：弹窗「是否继续等待？」
- [ ] **切后台/锁屏的处理**：生成要 60s，移动端用户很可能中途切走或锁屏。
  - 请求本身不受前后台影响（`fetch` 在后台仍会完成），关键是**回到前台时 UI 要能恢复**：用 `AppState` 监听，回前台后根据当前状态机阶段重绘进度，而不是停在切走时的旧帧
  - 若切走期间已生成成功：回前台直接展示结果（或轻提示「行程已生成好」）
  - 别依赖计时器在后台继续跑（RN 计时器在后台会被节流/暂停）

### 6.2 错误文案与操作

| 场景 | HTTP | 用户文案 | 操作 |
|------|------|----------|------|
| 无网络 | — | 无法连接，请检查 WiFi | 重试 |
| 未登录/过期 | 401 | 登录已过期，正在重新连接… | 自动重登（已有） |
| 生成格式失败 | 422 | 这次没生成好，换个说法试试 | 回表单 |
| AI 服务异常 | 502 | AI 暂时繁忙，请稍后重试 | 重试 |
| 请求超时 | — | 生成时间较长，请稍候或取消 | 继续/取消 |

### 6.3 行程「被削减」时的降级提示

> 技术方案 `enrichTrip` 会把查不到 poiId（或高德无此 ID）的非 transport 项**静默剔除**，避免幻觉坐标上地图。副作用：AI 排了 8 个点，回填后可能只剩 5 个，甚至某天变得很空——用户会以为「AI 偷懒」，其实是被后端过滤了。

- [ ] 后端在响应里带上剔除数量/明细（如 `droppedCount` 或 `warnings`），前端别只拿最终 items
- [ ] 生成结果页顶部轻提示：「有 N 个地点因信息不全未加入，可重新生成试试」
- [ ] 某天被删到过空（0～1 个点）：该天卡片显示占位「这天安排较少，点此重新生成本日」，而非展示一个空壳
- [ ] 全程剔除过多（如 >50%）：不要直接给残缺行程，回到 422 那套「没生成好，换个说法重试」

### 6.4 乐观 UI

- [ ] 点「保存行程」→ 列表立刻出现新项（骨架态），后台 `POST /api/trips`
- [ ] 失败则回滚并 Toast

### 6.5 离线体验（旅游场景刚需）

> 景区没信号、境外无流量、地铁里——**看已保存的行程不该依赖网络**。这是标题里「离线」二字的落点。

- [ ] **已保存行程本地缓存可读**：`GET /api/trips/:id` 成功后写入本地（AsyncStorage / React Query persist），断网时读缓存并顶部标「离线模式，显示的是上次同步的内容」
- [ ] 离线时**禁用/灰置**依赖网络的操作：重新生成、提交评价（评价可入本地队列，联网后补发）
- [ ] 地图离线：高德 SDK 离线瓦片成本高，第一版**不做离线地图**——离线时地图区域显示占位「地图需联网」，时间轴文字信息照常可读
- [ ] 跳转小红书/大众点评前判断网络，无网时提示「需要联网」而非跳出去白屏

### 6.6 弱网与后台通知

- [ ] **弱网预判**：`@react-native-community/netinfo` 检测到蜂窝/弱网时，生成前提示「建议连 WiFi，约需 1 分钟」
- [ ] **生成完成通知**（可选）：用户切后台且生成成功 → `expo-notifications` 本地通知「行程已生成」（需用户授权）
- [ ] **部分成功展示**：3 天行程只成功 2 天时，展示 2 天 + 失败天「点此重试本日」，而非整单 422

## 14. 怎么验证优化有效（轻量指标）

> 练手项目不必上专业埋点，但「改完有没有用」得有个数，否则优先级全靠拍脑袋。用 `console`/本地日志或一张免费 tier 的表先记着即可，重点是**养成看漏斗的习惯**。

**核心漏斗（一次完整任务）：**

```
打开 App → 点 chip/填表单 → 提交生成 → 生成成功 → 保存 → （体验后）评价
   每一步的流失，就是下一个要优化的地方
```

| 指标 | 怎么算 | 说明什么 | 关联优化 |
|------|--------|----------|----------|
| chip 点击率 | 点 chip 数 / 打开数 | 引导是否有效、新用户会不会用 | 3.1 快捷 chip |
| 生成成功率 | 200 数 / 提交生成数 | 后端/AI 稳定度、422 是否高频 | 6.2 错误、6.3 降级 |
| 生成中取消/流失率 | 取消或退出数 / 提交生成数 | 60s 等待是不是劝退点 | 6.1 分步 loading + 取消 |
| 行程被削减比例 | 有 dropped 的行程 / 总生成数 | poiId 剔除是否严重影响体验 | 6.3 降级提示 |
| 保存率 | 保存数 / 生成成功数 | 生成的行程用户认不认可 | 4.4 预览页质量 |
| 评价提交率 | 评价数 / 打开已保存行程数 | 闭环数据能不能攒起来 | 7.1 触发时机 |
| 首条消息发送率 | 发过至少 1 条的用户 / 打开数 | 引导是否让人敢开口 | 3.1 chip、首次引导 |
| 表单 abandonment | 打开表单未提交数 / 打开表单数 | 表单是否太难/太长 | 4.3、4.6 |
| 详情页停留时长 | 详情页平均停留秒数 | 行程是否值得看 | 5.1、5.5 |
| 编辑保存率 | 编辑后 PUT 成功数 / 进入编辑态数 | 手动微调是否有用 | 5.4 |

**用法：** 每做完一档优化（如 P0 快捷 chip），对比前后对应指标。别一次看全部，盯住当前阶段那 1～2 个即可。

## 16. 进阶交互方向（第三批：战略性，非近期排期）

> 前 15 章覆盖「核心任务链路 + 可靠性 + 打磨」。本章是**产品形态层面**的方向，多数属阶段 4 之后或「差异化亮点」，先立方向、留扩展点，不占用主线排期。
> 已在前文覆盖的相关点，这里只做指引不重复：触觉反馈见 `3.2`；会话持久化 / 停止生成 / 引用回复见 `8.4`；草稿恢复见 `5.6`；行程搜索收藏见 `5.3`；基础无障碍见第 `15` 章。

### 16.1 多模态输入：语音 + 拍照（阶段 4+）

**为什么值得做：** 旅游的本质是「人在移动、手不空、眼在看真实世界」，纯打字是最不契合的输入方式。这也是本项目相较普通聊天机器人的**场景红利**。

**① 拍照提问（建议先做，因为 Claude 原生支持视觉）**

- 场景：拍菜单问「哪个值得点」、拍景点海报问「怎么去」、拍街景问「这是哪」
- 交互：输入栏加 📷 按钮 → `expo-image-picker` 拍照/选图 → 压缩后随消息发给后端 → 后端把图片作为 `image` content block 传给 Claude
- 白话原理：Claude 能「看图」，图片和文字一起发过去，它就能就图作答，不需要额外接 OCR
- 坑：
  - 图片要**先压缩**（`expo-image-manipulator` 压到长边 ~1568px），否则上传慢、token 贵
  - 后端 `/api/chat` 要支持多模态入参（当前只收 `message` 文本，需扩展契约）
  - 成本比纯文本高，注意限频

**② 语音输入（长辈友好，但工程更重）**

- 场景：说「帮我规划成都三天」比打字快；**长辈**（大纲多次提到「带老人玩」）更愿意说不愿打
- 交互：输入栏加 🎤 按住说话 → 语音转文字 → 填进输入框（可编辑再发）
- 坑：
  - Claude **不做**语音识别（ASR），需另接服务（如系统原生 `expo-speech` 只做 TTS 不做 STT；STT 要接第三方或原生模块），工程量比拍照大
  - 先做「语音转文字填入框」，**不要**做全语音对话（还要 TTS 朗读回复，链路太长）
- ⚠️ 优先级：**拍照 > 语音**。拍照走 Claude vision 几乎零额外依赖，语音要引入 ASR，练手阶段先做拍照更划算

### 16.2 把 AI 本身当成「可交互对象」（阶段 4）

现有设计把 AI 当黑盒（输入→输出），只做了推荐理由和免责声明。但 AI 产品的体验核心是**让用户信任并能掌控这个 AI**。以下是前文尚未覆盖的部分：

- [ ] **能力发现**：用户不知道「能问什么、不能问什么」。除了 chip 示例，「我的」页或聊天顶部加一个「我能帮你做什么」说明（能查签证/天气/美食、能规划行程、不能订票不能付款）。降低「它到底行不行」的试探成本
- [ ] **聊天消息级反馈**：目前只有*行程*能评价（第 7 章），*聊天回复*答错了用户无处反馈。给 AI 气泡加轻量 👍/👎（长按菜单里即可），数据可后续用于改 prompt
- [ ] **数据来源标注**：行程里哪些是高德实时数据（评分/地址/坐标）、哪些是 AI 通用经验（推荐理由/攻略），用小角标或分区标出。**这是降低「它是不是在瞎编」焦虑的最有效手段**，也和第 10 章「可信任」原则一脉相承
- （停止生成、引用回复已在 `8.4`，此处不重复）

### 16.3 长辈模式（差异化亮点，阶段 4）

> 第 15 章的无障碍是「尊重系统设置」的**合规基线**；这里的长辈模式是**主动作为卖点的产品功能**——两者不同。既然「带老人玩」被反复当作使用场景，值得做成体系。

- [ ] **一键长辈模式**：「我的」页开关，打开后全局：更大字号、更高对比、更大按钮/触控区、更少信息密度（卡片只留时间+名称+导航）
- [ ] **简化流程**：长辈模式下规划表单只留「去哪+几天」，其余用默认值，减少决策
- [ ] **朗读行程**（可选，接 `expo-speech` TTS）：详情页「读给我听」，把当天安排念出来
- 白话价值：这块做好是**同类练手项目很少有的亮点**，且能学到移动端最容易被忽视的一课（真实适老化）
- 坑：长辈模式和系统动态字体会叠加，布局要在「系统大字 + 应用大字」双重放大下不崩（回归测试重点）

### 16.4 主动助手：从「有问必答」到「适时提醒」（阶段 5，远期）

> 它叫「助手」却完全被动——用户不问就什么都不发生。真正的 assistant 会在合适的时机主动出现。**这是产品的完全体，但对练手项目是奢侈品**，先记方向、不近期做。

- 出行前一天：本地通知「明天去成都，记得看看天气和 Day1 安排」
- 到饭点 + 定位在行程城市：轻提示「附近有你收藏的 XX，步行 5 分钟」
- 行程结束后：温和提醒补评价（配合第 7 章闭环）
- 坑：主动推送极易变打扰，**必须**默认克制 + 用户可全关；涉及定位要单独合规授权（第 10 章）。练手阶段可只做「出行前一天提醒」这一个最无争议的场景

### 16.5 反馈感官层（小成本高质感，穿插各阶段）

> 触觉已在 `3.2` 起步，这里把它升级为一个**成体系的反馈层**思路，供实现时统一处理。

- [ ] 关键节点统一触觉：发送、评价提交、**行程生成完成**（等了 60s，震一下告诉用户「好了」比只改屏幕更有效）
- [ ] 成功/失败用不同震动强度（`expo-haptics` 的 `Success` / `Error` 类型），形成肌肉记忆
- [ ] 过渡音效**默认关**（很多人静音使用），仅作为可选项
