# 阶段 3.5：评价闭环 —— 关键要点总结

> 本批目标：让用户**能对行程评价、评价能落库、进详情页能回显高亮**。
> 两条「反哺」（个人偏好注入 prompt、社区口碑注入搜索）**本批不做**，等有数据再接。

---

## 一、三个已确认的核心决策

1. **数据库用 SQLite 本地文件**（`node:sqlite`，Node 24 内置）
   - 零 npm 依赖、零原生编译（避开 `better-sqlite3` 要 Visual Studio 构建工具的 Windows 大坑）
   - 零注册、无境外访问风险、离线可用
2. **行程上云走「混合」**
   - 手机本地 AsyncStorage 仍是看行程的主力
   - 保存时**顺带上传**一份到服务端，拿回 `serverTripId` 存回本地，评价挂它上
   - 列表/详情页几乎不改，改动最小、风险最低
3. **本批只做「能评 / 能存 / 能回显」**，两条反哺后置

## 二、关键技术点 & 坑（踩错就白干）

### 坑 #1：Node 24 自带 SQLite
- `import { DatabaseSync } from 'node:sqlite'` 直接用，**不装任何包**
- `@types/node@24.13.3` 自带类型
- API：`db.prepare(sql).get(...)` 单行 / `.all(...)` 数组 / `.run(...)` 写入
- **本批零新依赖**

### 坑 #2：陈旧 token 的外键失败
- 手机缓存的旧 JWT 里的 `userId` 是老内存 Map 造的，**数据库 users 表里没这行**
- trips 表有 `user_id` 外键后，用旧 token 传行程会外键失败
- **解法**：换一次 `JWT_SECRET` → 旧 token 立即失效 → App 已跑通的「401 自动重登」自动换来一个 DB 里真实存在的 userId。用户无感知
- **注意**：换 secret 后必须**重启后端**才生效

### 坑 #3：SQLite 外键默认是关的
- `PRAGMA foreign_keys = ON` 必须开，否则 `ON DELETE CASCADE` / 外键校验**形同虚设**
- 在 `db.ts` 里建连接后立刻开

### 坑 #4：部分唯一索引（NULL≠NULL）
- 整程评价（`poi_id` 为 NULL）用普通 UNIQUE 拦不住重复——SQLite 里 `NULL ≠ NULL`
- **必须建两个部分唯一索引**：
  ```sql
  CREATE UNIQUE INDEX uniq_review_poi  ON reviews(user_id,trip_id,poi_id) WHERE poi_id IS NOT NULL;
  CREATE UNIQUE INDEX uniq_review_trip ON reviews(user_id,trip_id)        WHERE poi_id IS NULL;
  ```
- upsert 也要带 WHERE 谓词：`ON CONFLICT(user_id,trip_id) WHERE poi_id IS NULL DO UPDATE ...`

### 坑 #5：归属校验（防越权评价别人的行程）
- `POST /api/reviews` 和 `GET /api/reviews` 都必须先查 `trips` 表的 `user_id === req.userId`
- 不存在 → 404 `TRIP_NOT_FOUND`；不属于自己 → 403 `FORBIDDEN`

### 坑 #6：SQLite 无 boolean / 无数组 / 无 uuid
- `sentiment` 用 `INTEGER`(1/-1)，加 `CHECK (sentiment IN (1,-1))`
- `tags` 存 `TEXT`（`JSON.stringify` 存、`JSON.parse` 读，读时防御空串/坏 JSON）
- id 用 `randomUUID()`（`node:crypto`）→ `TEXT`
- 时间用 ISO 字符串 `TEXT`

### 坑 #7：本地保存必须先于上传（崩溃安全）
- `saveTrip` 顺序：**① 先本地存**（保证「保存」永远成功，哪怕断网/服务端挂了/App 被杀）→ **② 再上传**
- `uploadTrip` 是「尽力而为」：10 秒超时、失败返回 null、不抛错
- 失败时本地那份照样在，只是没有 `serverTripId` → 详情页显示「未同步，暂不能评价」

### 坑 #8：不碰 prompt（留给下一批）
- 本批零改 `claude.ts`
- 个人偏好要注入**用户消息**，不能进 system 缓存前缀（否则缓存前缀变，每次都 miss）
- 冷启动期没数据，反哺本来就是空的

### 坑 #9：Windows 终端 curl 中文乱码
- curl 带内联中文 → Windows GBK 终端编码 → 服务端按 UTF-8 解 → 乱码
- **这是测试工具的坑，不是后端的坑**：手机端用 `fetch` 发 UTF-8，不受影响
- 验证后端时用 Node `fetch` 驱动（保证 UTF-8），别用 curl 带中文

### 坑 #10：ESM / monorepo 导入约定
- 后端 `"type":"module"` + `NodeNext`：**本地 import 一律带 `.js`**（如 `'./db.js'`）
- 工作区包 `@travel/shared` **不带扩展名**
- `@travel/shared` 的 `package.json` `main`/`types` 直接指向 `./src/index.ts`（无构建步骤，直接消费源码）

### 坑 #11：循环依赖检查
- `tripStore` 现在要 import `api`（保存时调 `uploadTrip`）
- 已确认 `api.ts` **不反向 import `tripStore`** → 单向，安全

### 坑 #12：DB 文件路径 & gitignore
- 用 `import.meta.url` 推绝对路径（别用相对 cwd——`pnpm dev` 与 `node dist` 的 cwd 不同）
- `data/` 目录用 `mkdirSync(dir, {recursive:true})` 自动建
- `.gitignore` 忽略 `apps/server/data/`、`*.db`、`*.db-journal`、`*.db-wal`

## 六、数据形状速查

### 评价提交（`POST /api/reviews`）
```json
{ "tripId": "uuid", "poiId": null, "sentiment": -1, "tags": ["太赶"], "comment": "节奏太紧" }
```
- `poiId` 省略/null = 整程总评；有值 = 对某个 POI
- `sentiment`：1=赞 / -1=踩
- 后端 upsert：重复提交覆盖（靠两个部分唯一索引）

### 评价回显（`GET /api/reviews?tripId=`）
```json
{
  "tripReview": { "sentiment": -1, "tags": ["太赶"], "comment": "节奏太紧" },
  "poiReviews": { "B001": { "sentiment": 1, "tags": ["值得去"], "comment": null } }
}
```

### 标签库（来自 UX §7.2）
- 赞组：`性价比高` / `值得去` / `本地人多` / `交通方便`
- 踩组：`太赶` / `踩雷` / `排队久` / `名不副实`
