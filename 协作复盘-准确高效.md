# 协作复盘：哪些是准确高效的

> 背景：阶段 3.5（评价闭环）做完后，复盘这次对话里哪些做法判断准、没走弯路，哪些绕了。
> 用来以后保持好习惯、避免重复的低效操作。

---

## 一、准确（判断对了，没走弯路）

### 1. GBK 乱码没白改代码
**场景**：用 curl 带中文测后端，返回的标签乱码（`̫¥¬` 之类）。

**做对的地方**：没急着改后端代码，而是——
1. 先做 codepoint 分析，发现是真的 `U+fffd`（替换符），确认乱码真的发生了
2. 再用 Node `fetch` 复测同一接口，返回完全正常的 `["太赶","踩雷"]`
3. 判定：**这是 Windows 终端 curl 的 GBK 编码坑，不是后端的 bug，不用改代码**

**为什么准**：如果第一步就改代码，会去改一个本来没坏的东西，越改越乱。先证明「换个工具就正常」，就能定位问题在测试工具、不在代码。手机端用 `fetch` 发 UTF-8，根本不受影响。

---

### 2. saveTrip 顺序自己纠错
**场景**：写 `saveTrip` 时，一开始把 `uploadTrip`（上传到服务端）写在本地 AsyncStorage 保存**前面**。

**做对的地方**：自己发现这违反了计划里「本地先成功」的崩溃安全原则——如果上传失败，本地还没存，「保存」按钮就失败了，用户体验崩。于是改成：
1. 先本地存（永远成功）
2. 再上传（失败返回 null，不影响本地）
3. 顺手给 `uploadTrip` 补了 10 秒超时（之前漏了）

**为什么准**：没把这个 bug 留给你抓。崩溃安全顺序是这批的核心设计，自己对照原则检查一遍，比等测试暴露问题省事。

---

### 3. 换 JWT_SECRET 后先冒烟再交手机
**场景**：换了新的 `JWT_SECRET`（为了让旧 token 失效、触发自动重登），重启后端后。

**做对的地方**：没直接让你上手机测，而是先用 Node `fetch` 跑了一遍「登录 → 上传行程 → 评价 → 回显」全链路，确认新 secret 下后端没问题，再交给你测手机。

**为什么准**：这样万一手机出问题，就**只可能是前端**，排查范围窄一半。如果后端也有问题、前端也有问题，混在一起根本查不清。

---

### 4. 查进度时直接读数据库
**场景**：你问「检查项目进度」。

**做对的地方**：没只看任务清单（那些只代表「代码写完了」），而是查了 SQLite 数据库实际行数——发现已经有 4 份真实行程 + 1 条真实评价（👍 + 标签「本地人多」）。

**为什么准**：DB 里的真实数据 > 代码逻辑保证 > 任务清单勾选。这证明核心闭环你已经在手机上真跑通了，不是「代码应该能用」的纸上推演。

---

## 二、高效（省了来回）

### 5. 三个决策前置
**场景**：动手实现前。

**做对的地方**：用 AskUserQuestion 把三个关键决策一次问完——
1. 数据库选 SQLite 还是 Supabase
2. 行程上云走「混合」还是「全上云」
3. 本批范围只做「能评能存能回显」还是连反哺一起做

**为什么高效**：实现过程中不用停下来问，一气呵成。如果边做边问，节奏会被打断，还容易做到一半发现方向错了要回退。

---

### 6. 后端里程碑先于前端
**场景**：计划里定死「后端 curl 全绿，再动前端」。

**做对的地方**：后端 10 项 curl 测试（登录→上传→评价→upsert 覆盖→回显→单点→403 归属→404→401）全过了，才开始写前端。

**为什么高效**：bug 不会前后端混在一起查不清。后端绿了，前端出问题就是前端的事；后端没绿，前端写了也白写（调不通）。

---

### 7. 并行工具调用
**场景**：查进度、查服务器状态时。

**做对的地方**：把独立的信息查询放在一条消息里并行发——比如同时查端口占用 + 查数据库行数，一次拿全。

**为什么高效**：串行的话要等第一个结果回来再发第二个，白白多等。独立查询并行，省一半时间。

---

## 三、不够高效的几次（说实话）

### 8. Expo 启动失败两次才查端口
**场景**：启动 Expo dev server，第一次失败（非交互模式没法问「要不要换端口」），又起一次还失败。

**绕了的原因**：没先查 8081 端口是不是被旧的 Metro 占着。应该**起服务前先查端口占用**，而不是起了失败再查。

**教训**：起任何 dev server 前，先 `netstat -ano | grep :<端口>` 看一眼。

---

### 9. ipconfig 编码绕了三轮
**场景**：想查电脑局域网 IP 给手机连。

**绕的过程**：`ipconfig | grep IPv4` → 报「Binary file matches」（Windows 的 ipconfig 输出是 UTF-16，grep 当二进制了）→ 试 `iconv -f UTF-16LE` → 还是 binary → 最后才换 Node `os.networkInterfaces()`，一行搞定。

**绕了的原因**：Windows 命令行工具（ipconfig/tasklist）常输出 UTF-16，在 bash 管道里跟 grep/iconv 配合很差。

**教训**：Windows 上查系统信息（IP / 端口 / 进程），**优先用 Node 模块**（`os.networkInterfaces()`、`netstat`、`child_process`），别用 ipconfig。

---

## 四、一句话总结

| 类型 | 模式 |
|---|---|
| **准** | 先诊断根因再动手、自己纠错、用数据库证据说话 |
| **快** | 决策前置一次问完、后端先绿再动前端、并行查信息 |
| **避** | 起服务前先查端口、Windows 查信息用 Node 别用 ipconfig |

核心就一句：**遇到问题先证明再改，别凭感觉改；查状态用真实证据，别看清单推演。**
